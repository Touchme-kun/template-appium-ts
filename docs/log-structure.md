# Log Structure and Management Guide

## Overview

This document outlines the logging architecture, file locations, retention policies, and best practices for this mobile automation framework.

## Log Generation Locations

### 1. Winston Logger Outputs (Active Logs)

Located in `log/` directory — auto-created on first test run.

| File | Purpose | Format | Max Size | Retention |
|------|---------|--------|----------|-----------|
| `test-execution.log` | All application logs from tests | JSON | 10MB | 14 rotations (~140MB) |
| `error.log` | Errors only for quick diagnostics | JSON | 10MB | 14 rotations (~140MB) |
| `runs/run-{timestamp}.log` | Per-test-run logs for correlation | JSON | Unlimited | See below |

> **Note:** The log directory is `log/` (configured in `wdio.conf.ts` via `logPath`).

### 2. BrowserStack Local Output

Located in root: `local.log`

- **Purpose**: BrowserStack local tunnel server logs
- **Format**: Plain text
- **Auto-managed by**: BrowserStack local binary
- **Gitignore**: ✅ Properly ignored

### 3. Allure Report Artifacts

| Location | Purpose | Generated By | Gitignored |
|----------|---------|--------------|------------|
| `allure-results/` | Test metadata and attachments | WebdriverIO/Allure hooks | ✅ Yes |
| `allure-report/` | Rendered HTML report | `allure generate` command | ✅ Yes |
| `reports/screenshots/` | Failure screenshots | Test hooks (`afterTest`) | ✅ Yes |

## Log Lifecycle

### Active Logs (Default Retention: 30 days)

```
log/
├── test-execution.log        ← All logs, rotates at 10MB
├── error.log                 ← Errors only, rotates at 10MB
└── runs/
    ├── run-2026-06-09T10-15-30-000Z.log
    └── ...
```

**Rotation Policy:**
- File size limit: 10 MB per file
- Max files kept: 14 (oldest purged automatically by Winston)
- Total active space: ~140 MB

### Archive Logs (Retention: 90 days)

```
log/archive/                  ← Gitignored via logs/archive/ rule
├── 2026-06-01-archived-run.log
└── ...
```

## Managing Logs

### Automatic Management

Winston handles automatic rotation for `test-execution.log` and `error.log`:
- Triggered when file reaches 10 MB
- Old files named with numeric suffixes: `.1`, `.2`, etc.
- Oldest file deleted when count exceeds 14

### Manual Cleanup

Use the cleanup script to archive and manage logs:

```bash
# Show current log statistics
npm run logs:report

# Archive old logs (move logs older than 30 days to log/archive/)
npm run logs:archive

# Clean very old archives (delete archives older than 90 days)
npm run logs:clean

# Run all: archive + clean + report (recommended before a test run)
npm run logs:cleanup
```

Script source: `scripts/cleanup-logs.ts`

## Searching and Accessing Logs

### View Current Test Logs

```bash
# Windows (PowerShell)
Get-Content log\test-execution.log -Wait
Get-Content log\error.log -Wait

# macOS/Linux
tail -f log/test-execution.log
tail -f log/error.log
```

### Search for Issues

```bash
# Find all ERROR entries (bash)
grep ERROR log/test-execution.log

# Search for a specific test name
grep "TestName" log/test-execution.log
```

### Parse JSON Logs

```bash
# Pretty print JSON logs
cat log/test-execution.log | jq .

# Get only error-level messages
cat log/test-execution.log | jq 'select(.level == "error")'
```

## Log Levels

| Level | Usage | Enabled By |
|-------|-------|------------|
| `error` | Fatal failures, exceptions | Always |
| `warn` | Warnings, recoverable issues | `LOG_LEVEL=warn` or above |
| `info` | Test steps, major events | `LOG_LEVEL=info` (default) |
| `debug` | Detailed operation info | `LOG_LEVEL=debug` |

### Setting Log Level

```bash
# Via environment variable
LOG_LEVEL=debug npm run test:android

# Reduce verbosity for cleaner output
LOG_LEVEL=warn npm run test:android
```

## Logger Usage in Code

All logging must go through `Logger` from `src/utils/Logger.ts`. Never use `console.log`.

```typescript
import { Logger } from '../../utils/Logger';

// General messages
Logger.info('Test session starting...');
Logger.warn('Warning message', error as Error);
Logger.error('Operation failed', error as Error);
Logger.debug('Tapping at coordinates: (120, 340)');

// Test lifecycle
Logger.testStart('should login successfully');
Logger.testEnd('should login successfully', true, duration);
Logger.suiteStart('Smoke Test Suite');
Logger.suiteEnd('Smoke Test Suite', 5, 0, 1);

// Step tracing
Logger.step('Entering text');
Logger.action('Tapping button');

// Run correlation
Logger.info(`Run ID: ${Logger.getRunId()}`);
```

## Best Practices

### ✅ DO

- Use `Logger.*` for all logging — never `console.log`
- Use `Logger.step()` inside screen action methods for step tracing
- Use `Logger.action()` for user-facing actions (tap, enter text)
- Include context in error messages
- Run `npm run logs:cleanup` regularly
- Check `log/error.log` first when diagnosing failures

### ❌ DON'T

- Use `console.log` anywhere in `src/` or `configs/` — ESLint will warn
- Log sensitive data (PINs, passwords, API keys)
- Let `log/runs/` accumulate indefinitely — use `npm run logs:archive`
- Commit log files — `log/`, `*.log`, and `local.log` are all gitignored
